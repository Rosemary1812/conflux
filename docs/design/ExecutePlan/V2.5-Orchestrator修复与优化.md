# V2.5 Orchestrator 修复与优化

> 状态：计划待审  
> 基线：2026-05-30  
> 范围：Orchestrator 调度、Planner、Evaluator、群聊交互范式、总结输出  
> 目标：解决 V2.3–V2.4 中工程上过度简化、产品设计僵硬的问题，把 Orchestrator 从"所有消息都是任务"的单一范式，扩展为支持直接指派、纯对话、用户主导重试的多元交互。

---

## 背景

V2 群聊在 V1 单聊基础上快速扩展时，为了复用已有链路（Planner → Task → Dispatch → SSE），把所有群聊消息都等同于"任务编排请求"。这导致以下产品体验问题：

1. 用户无法在群聊中直接 @ 某个 Agent 单独提问或下发任务。
2. 用户无法与 Orchestrator 进行非任务类咨询/讨论。
3. Agent 执行任务时完全看不到群聊上下文，回复容易脱节。
4. 任务失败后机器自动重试，用户不知情、不可控。
5. 总结模板生硬，不体现任务之间的关系。

本计划汇总 `docs/state/TODO.md` 与 `docs/state/TOFIX.md` 中所有 Orchestrator 相关条目，并补充代码评审中发现的新问题，作为一次集中修复。

## 修复清单

### 模块 A：调度与执行（`service.ts` / `scheduler.ts` / `invoker.ts`）

| # | 优先级 | 问题 | 具体改法 | 涉及文件 |
|---|--------|------|----------|----------|
| A1 | P1 | 自动重跑机制（revision）粗糙且用户不可控 | ① 删除 `service.ts` 顶部 `revisionCount` Map；② 删除 `handleTaskCompleted` 中 `if (shouldRevise)` 分支（约 260–283 行）；③ 删除 `evaluator.ts` 中 `shouldRevise` 与 `createRevisionTask`；④ 删除 `scheduler.ts` 中废弃的 `createTasksFromPlan` | `service.ts`、`evaluator.ts`、`scheduler.ts` |
| A2 | P1 | 上游 task error 后下游 task 永久 pending，run 无法 finalize | `handleTaskCompleted` 中，若 task 状态为 `error`，遍历其下游依赖 task 并标记为 `cancelled`（附 `dependency_failed` 原因），触发 `areAllTasksTerminal` 完成汇总 | `service.ts` |
| A3 | P1 | Agent 执行任务时看不到群聊上下文（`contextMessages` 传空数组） | `invokeAgentForTask` 不再传空数组，改为传入相关群聊历史（用户原始消息 + 同轮已完成的依赖 task 摘要），总长度控制在 2000 字以内 | `invoker.ts`、`context.ts`（辅助提取历史） |
| A4 | P1 | Planner 返回的 task 若 alias 匹配失败，被静默丢弃 | `executePlan` 中 `if (!conversationAgentId) continue` 改为抛出明确错误，或插入 `status: "error"` 的 task 记录并推送 `task_status`，让用户知道分派失败 | `service.ts` |
| A5 | P1 | 废弃函数 `createTasksFromPlan` 仍留在仓库，插入空 FK 会炸 | 直接删除该函数 | `scheduler.ts` |
| A6 | P2 | Orchestrator 总结一次性出现，没有流式感 | `finalizeOrchestratorRun` 改为：先 insert 空消息（`status: "running"`）并推送占位事件 → 逐段更新 content 并推送 `message_delta` → 最终 `status: "done"` | `service.ts`、`stream-bus.ts` |

### 模块 B：计划与校验（`planner.ts` / `validator.ts`）

| # | 优先级 | 问题 | 具体改法 | 涉及文件 |
|---|--------|------|----------|----------|
| B1 | P1 | SYSTEM_PROMPT 被严重预设为 software/coding 场景 | 全文重写：去 coding 化身份定义、通用化 mode 描述、中性化规则语言（"Do NOT write code" → "Do NOT execute tasks yourself"），JSON schema 保持不变 | `planner.ts`、`TECH_DESIGN.md` |
| B2 | P1 | Planner 混淆 `platform`（`claude_code`）与 `alias`（`claude-code`） | ① Prompt 层：强调必须使用 roster 中的精确 alias，弱化/隐藏 platform 信息；② 代码容错层：`normalizePlan` 中 exact match 失败后尝试下划线↔连字符互换、忽略大小写等 fuzzy match | `planner.ts` |
| B3 | P2 | Mode 约束硬编码任务数量（compare 必须恰好 2 个等） | `validator.ts` 取消 `single_agent` / `compare` / `implement_review` 的 task 数量硬编码，改为只检查依赖关系和 alias 有效性 | `validator.ts` |
| B4 | P2 | Roster 中 `unavailable` 的 Agent 仍可能被派任务 | ① `validatePlan` 增加检查：task 的 assigneeAlias 对应 Agent 的 status 不能是 `unavailable`；② Planner prompt 加规则：不要把任务分配给 status=unavailable 的成员 | `validator.ts`、`planner.ts` |
| B5 | P2 | Planner 过度 clarify，模糊需求仍可能连续 2 轮 clarify | 微调 SYSTEM_PROMPT 中 clarify 触发条件，增加"直接做/开始/执行"等关键词权重；不设置硬轮数上限，由 Planner 自行判断信息充足度 | `planner.ts` |
| B6 | P2 | Clarify 机制不支持多轮追踪，且一次只能问一个问题 | ① `handleClarify` 从 DB 读取当前 run 的 `clarificationRound` 并递增，不再硬编码为 1；② Planner schema 的 `clarification_question` 扩展为 `clarification_questions: string[]`，支持一次性抛出多个问题；③ 不限制最大轮数 | `planner.ts`、`service.ts`、`types.ts` |
| B7 | P2 | `permission` 字段（readonly/editable）有名无实，Agent 根本不读 | Planner prompt 中不再要求输出 `permission` 字段；schema 保留但不再作为必填项，后续如需沙箱权限控制再在 adapter 层实现 | `planner.ts`、`types.ts` |

## 实施顺序建议

**Phase 1：阻塞性修复（P1，必须先做）**
- A1 删除自动重跑
- A2 上游失败下游自动取消
- A4 Task 静默丢弃改为报错
- A5 删除废弃代码
- B1 Planner 去 coding 化
- B2 Alias 混淆修复
- D1 群聊 @ 单 Agent 直接指派

**Phase 2：核心体验改进（P2，产品体验提升明显）**
- A3 Agent 运行时传入上下文
- A6 总结流式输出
- B3 Mode 约束取消硬编码
- B4 Roster status 参与调度
- B6 Clarify 多轮/多问
- C1 Evaluator 保守化
- C2/C3 总结模板按 mode 区分 + 显式失败说明

**Phase 3：功能扩展（P2，新能力）**
- D2 Orchestrator 纯对话模式
- D3 群聊 regenerate
- D4 Task 卡片 interaction 状态显示
- E1 Provider high risk 降级

**Phase 4：Polish**
- B5 Planner 过度 clarify 微调
- B7 Permission 字段弱化

## 明确不做（V2.5 范围内）

- 彻底删除 `permission` 字段的 schema 迁移（仅弱化，不动库表）。
- 多模态/图片理解（附件端到端已在 TODO P2，不归本计划）。
- 真正的 Agent 沙箱权限控制（V3 再评估）。
- 动态邀请 Agent 入群（V2 设计已明确不做）。

## 验收标准（整体）

1. `npm run typecheck`、`npm run build` 通过。
2. 单聊功能不退化（发送、流式、停止、重新生成、Approval/Choice）。
3. 群聊主链路不退化（初始化、Planner 分派、子 Agent 回复、Orchestrator 总结）。
4. 群聊中 @ 单个 Agent 可直接指派，该 Agent 直接回复，不产生 Orchestrator 规划消息和 task 卡片。
5. Planner 对 coding 和非 coding 请求均能返回合理 plan，不再默认带技术栈。
6. 上游 task error 后，下游 task 在 1 秒内自动变为 cancelled，Orchestrator 最终生成 summary。
7. 总结消息按 mode 呈现不同结构，失败任务被显式列出。
