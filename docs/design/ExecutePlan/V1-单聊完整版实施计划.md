# AgentHub V1 单聊完整版实施计划（新版原型对齐版）

## 一、总体目标

V1 按 `roadmap.md` 收敛：**单聊真实可用，群聊只做静态/模拟 UI**。

新版原型已经确定了一个重要交互：新建聊天不再先弹 Agent 选择器，而是直接进入空白会话。用户在首条消息里通过 `@Agent` 选择执行对象。

单聊规则：

- 新建聊天后进入空白单聊页。
- 首条消息必须且只能 `@` 一个 Agent，例如：`@claude-code 帮我 review 当前分支改动`。
- 首条消息发送成功后，该 Agent 被锁定为当前会话对象。
- 后续不能在同一单聊里切换 Agent；要换 Agent 必须新建聊天。

群聊规则：

- 新建群聊后进入空白群聊页。
- 首条消息可以 `@` 多个 Agent。
- UI 上自动展示 Orchestrator 加入。
- V1 不执行真实群聊调度，不调用 Orchestrator，不调用多个 Agent。

## 三、主要工作

- 搭 Next.js + TypeScript + SQLite + Drizzle 基础工程。
- 把新版 HTML 原型重写成 React 组件，不直接复用 HTML。
- 实现三栏布局：左侧会话列表、中间消息区、右侧上下文面板。
- 实现底部输入区四个固定入口：图片、附件、当前工作区、发送/停止。
- 发送按钮和停止按钮是同一个按钮的两种状态。
- 右栏支持收起和拖拽宽度。
- 设置从左下角按钮弹出 modal/sheet，不做独立业务路由。
- 实现单聊首条消息 `@Agent` 解析、校验、锁定。
- 实现消息落库、历史恢复、SSE 流式输出。
- 实现 Agent healthcheck 和不可用状态展示。
- 实现基础产物卡片、代码块、附件展示。
- 群聊页面使用 `lib/mock/group-conversation.ts`，只做静态预览和禁用发送。

## 四、阶段拆分

本计划不要一次性做完。后续 Agent 应按 phase 顺序推进，每个 phase 完成后先跑对应验收，再进入下一阶段。这样可以避免 UI、DB、SSE、子进程适配器同时改动导致问题难定位。

### Phase 0：工程基座

目标：先让正式应用能跑起来。

工作：

- 初始化 Next.js + TypeScript + Tailwind/shadcn 基础工程。
- 建立 `app/`、`components/`、`lib/` 目录骨架。
- 创建基础布局入口，不接 DB、不接真实 Agent。
- 保留 `prototypes/v1/` 作为参考，不直接复用 HTML。

验收：

- 本地开发服务能启动。
- 首页或会话页能看到三栏空壳。
- TypeScript/build 基础检查通过。

### Phase 1：前端原型 React 化

目标：把新版 HTML 原型转换成可维护的 React 组件，先全部使用 mock data。

工作：

- 实现 `AppShell`、`ConversationSidebar`、`MessageStream`、`Composer`、`ContextPanel`、`SettingsModal`。
- 对齐 `single-chat-new.html`：新建单聊进入空白页，首条消息通过 `@Agent` 绑定。
- 对齐 `single-chat.html`：消息流、产物卡片、右栏状态、底部 Composer。
- 对齐 `group-chat-new.html` 和 `group-chat.html`：群聊仅预览态。
- 对齐 `settings.html`：设置从左下角弹出 modal/sheet。
- Composer 固定四个入口：图片、附件、当前工作区、发送/停止。
- 右栏支持收起和拖拽宽度。

验收：

- 主要页面结构与新版原型一致。
- 新建单聊、新建群聊、设置弹层、右栏收起都能交互。
- 群聊页面明确显示 V1 仅 UI 或预览态。

### Phase 2：DB + 会话/消息 API

目标：让单聊会话、消息、Agent 锁定规则真实落库。

工作：

- 接入 Drizzle + SQLite。
- 定义 `agents`、`conversations`、`conversation_agents`、`messages`、`agent_runs`、`artifacts`。
- seed 内置 Agent：Claude Code、Codex、Hermes、OpenClaw。
- 实现会话列表、新建会话、会话详情、消息列表、发送消息 API。
- 实现单聊首条消息 mention 解析：
  - 没有 `@Agent`：拒绝。
  - 多个 `@Agent`：拒绝。
  - 一个有效 `@Agent`：写入 `conversation_agents` 并锁定会话。
  - 锁定后再 mention 其他 Agent：拒绝。

验收：

- 新建聊天后能发送首条 `@claude-code ...` 并落库。
- 刷新页面后会话、消息、锁定 Agent 都能恢复。
- 非法 mention 场景返回明确错误。
- 群聊发送不会触发真实 Agent 链路。

### Phase 3：SSE + 单聊运行状态

目标：先用 fake adapter 跑通流式链路和运行状态。

工作：

- 实现 `stream-bus`，把后端运行事件推给 `/api/conversations/:id/stream`。
- `POST message` 后创建 `agent_run`。
- 增加 fake adapter，模拟 `text_delta`、`run_status`、`message_done`、`message_error`。
- 前端接入 SSE，实时更新消息文本、运行状态和发送/停止按钮。
- 实现停止生成：调用 stop API，取消当前 run。

验收：

- 发送消息后能看到流式输出。
- 发送按钮在运行时切换为停止按钮。
- 停止后消息状态变为 cancelled。
- fake error 能在消息流中显示错误态。

### Phase 4：真实 Adapter 接入

目标：把 fake adapter 替换为真实外部 Agent 适配器，先保证单聊端到端。

工作：

- 定义 `AgentAdapter` 统一接口和 `AdapterRegistry`。
- 实现 `process-runner`：负责 spawn、stdout/stderr、退出码、AbortSignal、超时。
- 实现 `ClaudeCodeAdapter`，作为第一个真实端到端目标。
- 实现 `CodexAdapter`，作为第二个真实端到端目标。
- 实现 `HermesAdapter` 和 `OpenClawAdapter` 的 healthcheck/run 边界；可运行时接真实运行，不可用时返回明确不可用状态。
- 设置弹层展示四类 Agent 的 healthcheck 结果。

验收：

- Claude Code 至少真实端到端跑通：发送、流式回复、落库、刷新恢复。
- Codex 尽量真实端到端跑通；如果本机缺 CLI 或配置，必须展示准确失败原因。
- Hermes/OpenClaw 不允许用假成功；不可用时 UI 灰显或提示配置问题。
- 停止生成能终止真实进程或取消运行。

### Phase 5：产物与右栏

目标：让消息流和右栏形成完整 IM 工作台体验。

工作：

- 从 Agent 输出中提取代码块、文件路径、基础产物信息。
- 实现 `ArtifactCard`，支持预览/下载入口的 UI 状态。
- 右栏展示当前 Agent、运行状态、进度、产出文件。
- 实现重新生成：删除或标记旧 Agent 回复，重新调用当前锁定 Agent。
- 附件和图片先完成 UI/元数据链路，实际传给各 Agent 的格式按适配器能力处理。

验收：

- Agent 回复中的代码块能正确显示。
- 产物卡片和右栏产出文件一致。
- 重新生成能调用同一个锁定 Agent。
- 刷新后产物仍能恢复。

### Phase 6：群聊静态收口 + 文档

目标：明确 V1 边界，避免误以为群聊已经真实接入。

工作：

- 群聊新建页和群聊预览页使用 `lib/mock/group-conversation.ts`。
- 群聊 Composer 保持禁用或 mock-only，不调用真实 adapters。
- 补充 `TECH_DESIGN.md`：架构、数据模型、Adapter 契约。
- 补充 `API_CONTRACT.md`：HTTP API 和 SSE 事件。
- 补充 `REVIEW_CHECKLIST.md`：V1 验收清单。
- 未及时修复的小问题记录到 `docs/state/TOFIX.md`。

验收：

- 群聊页面可演示产品形态，但清楚标注 V1 仅 UI。
- API 层不会让 group conversation 调用真实 Orchestrator。
- 文档能指导下一个 Agent 从任一 phase 继续。

## 五、适配器接入

适配器是 Conflux 调用外部 Agent 的翻译层。业务层只认统一接口，不直接关心 Claude Code、Codex、Hermes、OpenClaw 的具体命令。

统一契约：

```ts
interface AgentAdapter {
  platform: "claude_code" | "codex" | "hermes" | "openclaw";

  healthcheck(): Promise<{
    ok: boolean;
    message: string;
  }>;

  run(params: {
    runId: string;
    conversationId: string;
    workspacePath: string;
    messages: AdapterMessage[];
    attachments: AdapterAttachment[];
    signal: AbortSignal;
  }): AsyncIterable<AgentEvent>;
}
```

各平台实现：

- `ClaudeCodeAdapter`：基于 **Claude Agent SDK** 调用本机 Claude Code（`profile: builtin`，不覆盖 System Prompt）；详见 `docs/design/TECH_DESIGN.md`。
- `CodexAdapter`：调用本机 Codex CLI，作为第二个真实跑通目标。
- `HermesAdapter`：实现同样的 healthcheck/run；如果结构化流不稳定，V1 允许先用 stdout 文本流 fallback。
- `OpenClawAdapter`：一等适配器，不做假数据；本机不可用时显示未安装/未配置。

统一事件：

```ts
type AgentEvent =
  | { type: "text_delta"; delta: string }
  | { type: "artifact_created"; artifact: ArtifactPayload }
  | { type: "run_status"; status: string }
  | { type: "message_done" }
  | { type: "message_error"; error: string }
  | { type: "message_cancelled" };
```

## 六、验收标准

单聊必须通过：

- 新建聊天进入空白页。
- 首条消息没有 `@Agent` 时提示错误。
- 首条消息 `@` 多个 Agent 时提示错误。
- 首条消息 `@claude-code` 后锁定 Claude Code。
- 后续消息不用重复 `@Agent` 也能继续对话。
- 后续尝试切换 Agent 会被拒绝。
- 至少 Claude Code 端到端跑通：发送、流式回复、落库、刷新恢复。
- Codex、Hermes、OpenClaw 至少完成 healthcheck 和适配器边界；可用环境下端到端运行。
- 停止生成能中断正在运行的 Agent。
- 重新生成能重新调用当前锁定 Agent。
- 右栏能展示 Agent 状态、进度、产出文件。
- 设置弹层能展示各 Agent 连接状态。
- 群聊页面明确标注 V1 仅 UI，不触发真实调度。

## 七、默认假设

- V1 不做 Electron。
- V1 不做真实 Orchestrator。
- V1 不做 `/agent-creator`、`/skill-creator`。
- V1 不做自建 Agent 执行。
- `Agent.description` 只作为 UI 展示和未来编排提示，不参与执行。
- 当前工作区 pill 表示 Agent 的活动目录和产物输出目录。
- 原型中的 OpenCode 视为参考项，本计划按用户要求优先保证 Claude Code、Codex、Hermes、OpenClaw。
