# AgentHub V1 单聊完整版实施计划（新版原型对齐版）

## 一、总体目标

V1 按 `roadmap.md` 收敛：**单聊真实可用，群聊只做静态/模拟 UI**。

新版原型已经确定了一个重要交互：新建聊天不再先弹 Agent 选择器，而是直接进入空白会话。用户在首条消息里通过 `@Agent` 选择执行对象。

单聊规则：

- 新建聊天后进入空白单聊页。
- 首条消息必须且只能 `@` 一个 Agent，例如：`@claude-code 帮我 review 当前分支改动`。
- 首条消息发送成功后，该 Agent 被锁定为当前会话对象。
- 后续不能在同一单聊里切换 Agent；要换 Agent 必须新建聊天。

群聊规则：

- 新建群聊后进入空白群聊页。
- 首条消息可以 `@` 多个 Agent。
- UI 上自动展示 Orchestrator 加入。
- V1 不执行真实群聊调度，不调用 Orchestrator，不调用多个 Agent。

## 三、主要工作

- 搭 Next.js + TypeScript + SQLite + Drizzle 基础工程。
- 把新版 HTML 原型重写成 React 组件，不直接复用 HTML。
- 实现三栏布局：左侧会话列表、中间消息区、右侧上下文面板。
- 实现底部输入区四个固定入口：图片、附件、当前工作区、发送/停止。
- 发送按钮和停止按钮是同一个按钮的两种状态。
- 右栏支持收起和拖拽宽度。
- 设置从左下角按钮弹出 modal/sheet，不做独立业务路由。
- 实现单聊首条消息 `@Agent` 解析、校验、锁定。
- 实现消息落库、历史恢复、SSE 流式输出。
- 实现 Agent healthcheck 和不可用状态展示。
- 实现基础产物卡片、代码块、附件展示。
- 群聊页面使用 `lib/mock/group-conversation.ts`，只做静态预览和禁用发送。

## 四、阶段拆分

本计划不要一次性做完。后续 Agent 应按 phase 顺序推进，每个 phase 完成后先跑对应验收，再进入下一阶段。这样可以避免 UI、DB、SSE、子进程适配器同时改动导致问题难定位。

### Phase 0：工程基座

目标：先让正式应用能跑起来。

工作：

- 初始化 Next.js + TypeScript + Tailwind/shadcn 基础工程。
- 建立 `app/`、`components/`、`lib/` 目录骨架。
- 创建基础布局入口，不接 DB、不接真实 Agent。
- 保留 `prototypes/v1/` 作为参考，不直接复用 HTML。

验收：

- 本地开发服务能启动。
- 首页或会话页能看到三栏空壳。
- TypeScript/build 基础检查通过。

### Phase 1：前端原型 React 化

目标：把新版 HTML 原型转换成可维护的 React 组件，先全部使用 mock data。

工作：

- 实现 `AppShell`、`ConversationSidebar`、`MessageStream`、`Composer`、`ContextPanel`、`SettingsModal`。
- 对齐 `single-chat-new.html`：新建单聊进入空白页，首条消息通过 `@Agent` 绑定。
- 对齐 `single-chat.html`：消息流、产物卡片、右栏状态、底部 Composer。
- 对齐 `group-chat-new.html` 和 `group-chat.html`：群聊仅预览态。
- 对齐 `settings.html`：设置从左下角弹出 modal/sheet。
- Composer 固定四个入口：图片、附件、当前工作区、发送/停止。
- 右栏支持收起和拖拽宽度。

验收：

- 主要页面结构与新版原型一致。
- 新建单聊、新建群聊、设置弹层、右栏收起都能交互。
- 群聊页面明确显示 V1 仅 UI 或预览态。

### Phase 2：DB + 会话/消息 API

目标：让单聊会话、消息、Agent 锁定规则真实落库。

工作：

- 接入 Drizzle + SQLite。
- 定义 `agents`、`conversations`、`conversation_agents`、`messages`、`agent_runs`、`artifacts`。
- seed 内置 Agent：Claude Code、Codex、Hermes、OpenCode。
- 实现会话列表、新建会话、会话详情、消息列表、发送消息 API。
- 实现单聊首条消息 mention 解析：
  - 没有 `@Agent`：拒绝。
  - 多个 `@Agent`：拒绝。
  - 一个有效 `@Agent`：写入 `conversation_agents` 并锁定会话。
  - 锁定后再 mention 其他 Agent：拒绝。

验收：

- 新建聊天后能发送首条 `@claude-code ...` 并落库。
- 刷新页面后会话、消息、锁定 Agent 都能恢复。
- 非法 mention 场景返回明确错误。
- 群聊发送不会触发真实 Agent 链路。

### Phase 3：SSE + 单聊运行状态

目标：先用 fake adapter 跑通流式链路和运行状态。

工作：

- 实现 `stream-bus`，把后端运行事件推给 `/api/conversations/:id/stream`。
- `POST message` 后创建 `agent_run`。
- 增加 fake adapter，模拟 `text_delta`、`run_status`、`message_done`、`message_error`。
- 前端接入 SSE，实时更新消息文本、运行状态和发送/停止按钮。
- 实现停止生成：调用 stop API，取消当前 run。

验收：

- 发送消息后能看到流式输出。
- 发送按钮在运行时切换为停止按钮。
- 停止后消息状态变为 cancelled。
- fake error 能在消息流中显示错误态。

### Phase 4：真实 Adapter 接入

目标：把 fake adapter 替换为真实外部 Agent 适配器，先保证单聊端到端。

工作：

- 定义 `AgentAdapter` 统一接口和 `AdapterRegistry`。
- 实现 `process-runner`：负责 spawn、stdout/stderr、退出码、AbortSignal、超时。
- 实现 `ClaudeCodeAdapter`，作为第一个真实端到端目标。
- 实现 `CodexAdapter`，作为第二个真实端到端目标。
- 实现 `HermesAdapter` 和 `OpenCodeAdapter` 的 healthcheck/run 边界；可运行时接真实运行，不可用时返回明确不可用状态。
- 设置弹层展示四类 Agent 的 healthcheck 结果。

验收：

- Claude Code 至少真实端到端跑通：发送、流式回复、落库、刷新恢复。
- Codex 尽量真实端到端跑通；如果本机缺 CLI 或配置，必须展示准确失败原因。
- Hermes/OpenCode 不允许用假成功；不可用时 UI 灰显或提示配置问题。
- 停止生成能终止真实进程或取消运行。

### Phase 5：产物与右栏

目标：让消息流和右栏形成完整 IM 工作台体验；补齐 V1 单聊里「工作区、附件、消息操作、产物展示、右栏 Terminal」五条主线。

**产品形态（V1 已确认）**

- 用户通过本机 CLI 启动产品（如 `npx agenthub` / `npm start`），在本机起一个 Node 服务并自动打开浏览器。
- 浏览器只是 UI 壳，通过 `localhost` 连接本机服务；Agent adapter、SQLite、PTY 终端均跑在用户电脑上。
- **不做**纯云端静态站（无本机 Node 进程）下的 Terminal / Agent CLI 能力。
- **不做** Electron；不尝试从浏览器拉起系统 Terminal.app / Windows Terminal 独立窗口。

#### 5.2 概念区分：代码块显示 vs 产物识别

这两件事不要混为一谈：

- **代码块显示（必做）**：Agent 回复里的 ````lang ... ```` 做 Markdown 渲染 + Shiki 高亮。这是基础 IM 体验，验收项「代码块能正确显示」指这个。
- **产物识别（按需分档）**：当 Agent 在磁盘生成文件或在文本中提到路径时，写入 `artifacts` 表，驱动消息流 **产物卡片** 与右栏 **产出文件列表**。

产物数据来源优先级：

1. adapter 显式发出 `artifact_created` 事件（首选，少靠猜测）；
2. 扫描当前会话 `workspace_path` 下新增/变更文件（本地 demo 可用）；
3. 从 Agent 文本中提取文件路径（兜底，易误判，仅作补充）。

若 Phase 5 时间紧，可先完成 P0（工作区、附件、代码块、重新生成），产物卡片与识别逻辑可降为 P1。

#### 5.3 工作区（P0 详述）

- 数据：`conversations.workspace_path`（或等价字段），创建会话时可默认上次使用或启动时的目录。
- UI：Composer「当前工作区」pill 改为可点击，选择本机目录；顶栏 meta 同步显示真实路径。
- 运行：`AdapterRunParams.workspacePath` 与会话字段一致；Claude `--add-dir`、Codex `-C`、OpenCode `--dir`、Hermes `cd` 均使用该路径。
- Terminal：PTY 会话 `cwd` 与 `workspace_path` 一致，保证 Agent 与手动命令在同一目录。
- 验收：切换工作区后发消息、开 Terminal，均在选定目录下执行；刷新后会话仍记住该路径。

#### 5.4 附件与图片（P0 详述）

- 前端：图片/附件按钮接入文件选择；发送前预览；与文本消息一并提交。
- 后端：文件存本地（如 `data/attachments/` 或 PRD 约定的 `~/.conflux/`）；`messages` 或关联表保存 metadata（文件名、MIME、大小、存储路径）。
- 消息流：图片缩略图、附件卡片（图标 + 下载）。
- Adapter：能传则传（按 Claude/Codex 等平台能力）；不能传时至少 UI 可见、用户可下载，不假装已传给 Agent。

#### 5.6 右栏 Terminal 技术方案（P1，已确认）

**目标**：用户在浏览器右栏获得与本机 shell 等价的交互，cwd 为当前会话工作区。

**架构**

```
用户执行: npx agenthub  (或等价 npm 全局/本地命令)
    ↓
本机 Node 进程启动（提供 HTTP + WebSocket + PTY）
    ↓
自动打开浏览器 → http://localhost:<port>
    ↓
浏览器右栏 xterm.js ←WebSocket→ node-pty (PowerShell / bash)
                              cwd = conversation.workspace_path
```

**实现要点**

- 前端：`xterm.js` + `@xterm/addon-fit`（可选）；挂载于右栏 Terminal 视图。
- 后端：`node-pty` 创建 PTY；WebSocket 路由（可与 Next 自定义 server 或独立 ws 端点）；字节流双向转发。
- 绑定：每个 Terminal 会话对应当前单聊的 `workspace_path`；切换会话时重建或切换 PTY cwd。
- 交互：支持常规命令、`Ctrl+C` 中断、基础彩色输出；不承诺 vim/htop 等全屏 TUI 完美体验。

**明确不做**

- 从浏览器 spawn 本机 GUI 终端（Terminal.app、Windows Terminal、iTerm）
- 无本机 Node 进程时的云端 Terminal（纯 Vercel 静态部署）
- Electron / Tauri 壳

**与 Agent 的关系**

Terminal 与 Agent adapter 并列：Agent 通过 CLI adapter 自动执行；Terminal 供用户在同一工作区手动执行命令。二者共享 `workspace_path`，互不替代。

## 五、适配器接入

适配器是 Conflux 调用外部 Agent 的翻译层。业务层只认统一接口，不直接关心 Claude Code、Codex、Hermes、OpenCode 的具体命令。

统一契约：

```ts
interface AgentAdapter {
  platform: "claude_code" | "codex" | "hermes" | "opencode";

  healthcheck(): Promise<{
    ok: boolean;
    message: string;
  }>;

  run(params: {
    runId: string;
    conversationId: string;
    workspacePath: string;
    messages: AdapterMessage[];
    attachments: AdapterAttachment[];
    signal: AbortSignal;
  }): AsyncIterable<AgentEvent>;
}
```

各平台实现：

- `ClaudeCodeAdapter`：基于 **Claude Agent SDK** 调用本机 Claude Code（`profile: builtin`，不覆盖 System Prompt）；详见 `docs/design/TECH_DESIGN.md`。
- `CodexAdapter`：调用本机 Codex CLI，作为第二个真实跑通目标。
- `HermesAdapter`：实现同样的 healthcheck/run；如果结构化流不稳定，V1 允许先用 stdout 文本流 fallback。
- `OpenCodeAdapter`：调用本机 OpenCode CLI，V1 先按 `opencode run --format json --dir <workspace> <prompt>` 接入；本机不可用时显示未安装/未配置。

统一事件：

```ts
type AgentEvent =
  | { type: "text_delta"; delta: string }
  | { type: "artifact_created"; artifact: ArtifactPayload }
  | { type: "run_status"; status: string }
  | { type: "message_done" }
  | { type: "message_error"; error: string }
  | { type: "message_cancelled" };
```

## 六、验收标准

单聊必须通过：

- 新建聊天进入空白页。
- 首条消息没有 `@Agent` 时提示错误。
- 首条消息 `@` 多个 Agent 时提示错误。
- 首条消息 `@claude-code` 后锁定 Claude Code。
- 后续消息不用重复 `@Agent` 也能继续对话。
- 后续尝试切换 Agent 会被拒绝。
- 至少 Claude Code 端到端跑通：发送、流式回复、落库、刷新恢复。
- Codex、Hermes、OpenCode 至少完成 healthcheck 和适配器边界；可用环境下端到端运行。
- 停止生成能中断正在运行的 Agent。
- 重新生成能重新调用当前锁定 Agent。
