# 群聊同类型多实例隔离修复计划

## 背景

本计划承接 `docs/state/TOFIX.md` 中 2026-06-06 的 P1 待修复项。该问题不是单一 bug，而是“群聊中重复使用同一种真实 Agent”后，身份、展示、寻址、底层外部 session 和子 Agent 角色约束没有完全对齐。

典型场景：

```text
用户：@claude @claude 并行审查这个方案
预期：群聊中出现两个可区分的 Claude Code 实例，各自接收任务、各自回复、各自续接自己的底层 Claude session。
```

当前数据层已经能生成两个 `conversation_agents` 实例，例如 `claude-code` 与 `claude-code-2`。但运行体验还没有完全按 `conversation_agent_id` 作为群聊成员身份来收敛。

## 当前未完成事项

### 1. Orchestrator 分工说明缺失

当前 execute 分支创建 task 后会直接启动子 Agent，消息流里可能看不到清晰的“我会把任务分给谁做什么”的 Orchestrator 分工说明。

影响：

- 用户看到子 Agent 直接开始回复，不知道调度依据。
- 多 Agent 并行时，消息顺序和任务归属不够清楚。

目标：

- execute 阶段进入子 Agent dispatch 前，先写入一条 Orchestrator 普通消息。
- 该消息通过 SSE 进入前端消息流。
- 消息内容只说明分工，不代替子 Agent 输出。

### 2. 同类型实例展示不可区分

重复 `@claude @claude` 后，数据层 alias 是 `claude-code` / `claude-code-2`，但 UI 气泡、右栏参与者和任务上下文仍可能都显示基础名称 `Claude Code`。

影响：

- 用户无法判断哪个气泡属于第一个 Claude Code，哪个属于第二个 Claude Code。
- stop、重新分派、查看任务状态时容易操作错实例。

目标：

- 同 slug 多实例生成可读展示名，例如 `Claude Code 1`、`Claude Code 2`。
- UI 所有群聊成员位置优先显示 `conversation_agents.display_name`，并保留 alias 作为可输入地址。
- 消息气泡、右栏参与上下文、任务分派列表保持同一身份命名。

### 3. 群聊成员 mention 寻址不清楚

初始化后，后端只按 roster alias 解析 mention。用户如果看到展示名 `Claude Code 2`，自然可能尝试输入 `@Claude Code 2`，但当前 mention 解析只支持无空格 token，容易报“不在当前群聊”。

决策：

- V2.6 先将 alias 作为唯一可输入地址，例如 `@claude-code`、`@claude-code-2`。
- 展示名用于视觉区分，不作为自由文本 mention 的主解析对象。
- UI 必须让 alias 可见，必要时以成员 chip、tooltip 或复制入口暴露。

目标：

- 后续发送 `@claude-code-2 ...` 能稳定指向第二个实例。
- UI 不引导用户输入 `@Claude Code 2` 这类带空格展示名。
- 如果用户输入未知 alias，错误提示要说明“请使用当前群聊成员 alias”，并列出可用 alias。

### 4. 外部 Claude session 未按实例隔离

这是本计划里最关键的数据一致性问题。

当前 `agent_runs` 已经记录 `conversation_agent_id`，但 `agent_external_sessions` 仍按以下维度保存和查找：

```text
conversation_id + agent_id + platform
```

这会导致同一个群聊里的两个 Claude Code 实例共享同一条外部 session 记录。谁最后保存 session id，下一轮同类型实例就可能 resume 到谁的底层 Claude session。

影响：

- `Claude Code 1` 和 `Claude Code 2` 在 AgentHub 内部看起来是两个成员。
- 但 Claude Code SDK 底层上下文可能被共用或覆盖。
- 子 Agent 的内部历史、工具状态和上下文归属不严格。

目标：

- `agent_external_sessions` 增加 `conversation_agent_id`。
- 群聊子 Agent 保存和读取外部 session 时必须使用 `conversation_agent_id`。
- 新 key 语义为：

```text
conversation_id + conversation_agent_id + agent_id + platform
```

- 单聊保持兼容：已有历史 session 不丢失，新 run 尽量使用单聊锁定的 `conversation_agent_id`；缺失时回退旧键。

### 5. 子 Agent 角色约束不足

子 Agent 收到 Orchestrator 分派任务后，prompt 里缺少明确约束：它应该只完成当前被分派任务，不应继续扮演 Orchestrator，也不应声称要再启动多个 Agent。

影响：

- Claude Code 作为子 Agent 时，可能自称会继续拆分、调度或启动其他 Agent。
- 群聊职责边界混乱。

目标：

- Orchestrator invoker 为子 Agent 注入明确角色约束。
- 子 Agent prompt 必须包含：
  - 你是被分派的执行 Agent，不是 Orchestrator。
  - 只完成当前 task。
  - 不要声明会再调度其他 Agent。
  - 如需要用户决策，使用已有 Approval / Choice 机制。

## 目标契约

本次修复后，群聊运行身份统一以 `conversation_agent_id` 为准：

| 概念 | 用途 |
| --- | --- |
| `agent_id` | 基础 Agent 产品或适配器，例如 Claude Code |
| `conversation_agent_id` | 群聊中的具体成员实例，例如第一个 Claude Code、第二个 Claude Code |
| `alias` | 用户可输入的成员地址，例如 `claude-code-2` |
| `display_name` | UI 展示名称，例如 `Claude Code 2` |
| `external_session_id` | 底层 Claude Code / OpenCode 等外部运行时 session |

核心规则：

1. 群聊任务分派、消息作者、运行记录、交互卡片、stop 操作都指向 `conversation_agent_id`。
2. 同类型多实例共享底层 platform 配置和同一个 `workspace_path`，但不共享外部 session。
3. Orchestrator 负责规划、分派、汇总；子 Agent 负责执行自己的 task。
4. 消息流必须能看出“谁被分配了什么、谁正在回复、谁完成了什么”。

## 实施步骤

### Phase 0：复现和基线确认

目标：先确认当前问题在本地可复现，避免修复方向跑偏。

检查项：

- 新建群聊发送 `@claude @claude ...`。
- 数据库中存在两条 `conversation_agents`，alias 分别为 `claude-code` / `claude-code-2`。
- `agent_runs.conversation_agent_id` 能区分两个 run。
- `agent_external_sessions` 只有 `conversation_id + agent_id + platform` 粒度，不能区分两个实例。

涉及文件：

- `lib/agents/mention.ts`
- `lib/conversations/service.ts`
- `lib/conversations/runs.ts`
- `lib/db/schema.ts`
- `lib/db/client.ts`

### Phase 1：实例展示与 alias 统一

目标：先让用户明确看到“两个 Claude Code 是两个成员”。

改动：

- roster 初始化时，为同 slug 多实例生成稳定 `display_name`。
- `listConversationAgents` 返回 `displayName` 和 `alias`。
- `MessageBubble` 群聊气泡使用 `authorConversationAgentId` 对应的 display name。
- `ContextPanel` 和任务分派区域统一使用同一套成员展示。
- Composer 或成员列表暴露可输入 alias，避免用户误输入带空格展示名。

涉及文件：

- `lib/conversations/service.ts`
- `components/chat/MessageBubble.tsx`
- `components/chat/Composer.tsx`
- `components/context/ContextPanel.tsx`

验收：

- `@claude @claude ...` 后，前端能看到 `Claude Code 1` / `Claude Code 2`。
- UI 同时能看到或复制 `claude-code` / `claude-code-2`。
- 后续输入 `@claude-code-2 ...` 能指向第二个成员。

### Phase 2：Orchestrator 分工消息

目标：让 execute 阶段的调度过程在消息流可见。

改动：

- `executePlan` 创建 task 后、dispatch 子 Agent 前写入 Orchestrator 消息。
- 消息内容基于 plan task 列表生成简短分工说明。
- 通过现有 SSE 事件发布 `message_replace` 或等价消息事件，让前端立即显示。

涉及文件：

- `lib/orchestrator/service.ts`
- `lib/conversations/stream-bus.ts`（如需补事件类型）

验收：

- 消息顺序为：用户消息 -> Orchestrator 分工说明 -> 子 Agent 气泡 -> Orchestrator 汇总。
- Orchestrator 分工消息不吞掉或代述子 Agent 最终输出。

### Phase 3：外部 session 按实例隔离

目标：修复底层 Claude session 共享或覆盖问题。

改动：

- `agent_external_sessions` 增加 `conversation_agent_id`。
- migration 需要处理旧表唯一约束，必要时重建表。
- 新唯一约束按实例隔离：

```text
conversation_id + conversation_agent_id + agent_id + platform
```

- `getExternalSession` / `saveExternalSession` 接收 `conversationAgentId`。
- `drainAgentRun` 调 adapter 时按当前 run 的 `conversationAgentId` 查找和保存 session。
- 兼容旧数据：如果 `conversationAgentId` 缺失，允许回退旧 session 行，但新群聊 run 不再写旧粒度。

涉及文件：

- `lib/db/schema.ts`
- `lib/db/client.ts`
- `lib/conversations/runs.ts`
- `lib/adapters/types.ts`（如需把实例信息显式传给 adapter）

验收：

- 两个 Claude Code 实例各自保存不同的 `agent_external_sessions` 行。
- 第二个实例保存 session 后，不会覆盖第一个实例的 session。
- 后续 `@claude-code` 和 `@claude-code-2` 分别 resume 自己的外部 session。
- 单聊历史会话不因 migration 丢失续接能力。

### Phase 4：子 Agent 角色约束

目标：防止子 Agent 自发扮演 Orchestrator。

改动：

- `invokeAgentForTask` 或 `startAgentRun` 生成子 Agent task prompt 时注入角色边界。
- Claude Code adapter 的 system prompt 保持通用 AgentHub 行为，群聊子 Agent 约束由 invoker/taskPrompt 注入。
- prompt 中明确要求子 Agent 只执行当前任务，不再拆分或调度其他 Agent。

涉及文件：

- `lib/orchestrator/invoker.ts`
- `lib/conversations/runs.ts`
- `lib/adapters/claude-code.ts`（如最终选择 adapter 级补充）

验收：

- 子 Agent 回复围绕被分派任务。
- 子 Agent 不再声明“我会启动多个 Agent”或继续扮演调度者。
- 需要用户判断时仍走 Approval / Choice。

### Phase 5：验证与回归

基础命令：

```powershell
npm run typecheck
npm run build
```

人工验收场景：

1. 新建群聊，发送 `@claude @claude 并行分析这个项目的 UI 和 API 风险`。
2. 确认 roster 为两个实例：`claude-code` / `claude-code-2`。
3. 确认消息流顺序：用户消息 -> Orchestrator 分工 -> 两个子 Agent 回复 -> Orchestrator 汇总。
4. 确认两个子 Agent 气泡、右栏参与者、任务分派均能区分实例。
5. 后续发送 `@claude-code-2 继续补充 API 风险`，只派给第二个实例。
6. 查询数据库，确认两个实例的 `agent_external_sessions` 不互相覆盖。
7. 子 Agent 回复不再自称要继续调度其他 Agent。

## 暂不纳入本次修复

- 不做真正的多 workspace 隔离；同一群聊仍共享 `conversations.workspace_path`。
- 不做富文本 mention 组件；本次先使用 alias 作为唯一可输入地址。
- 不改 Orchestrator 的核心规划算法，只修复身份、展示、session 和角色边界。
- 不重写 adapter 主循环。

## 完成标准

本计划完成后，`TOFIX.md` 中 2026-06-06 的 P1 项可以移动到已做，并补充验证结果。

最低完成标准：

- `npm run typecheck` 通过。
- `npm run build` 通过。
- `@claude @claude ...` 群聊场景能稳定区分两个实例。
- 两个实例的外部 session 不互相覆盖。
- 子 Agent 只围绕被分派任务回复。
