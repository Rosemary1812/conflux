# V1.5 交互桥接：Approval 与选项（V1→V2 前置）

## 1. 阶段定位

V1（Phase 0–6）已收口单聊主链路与群聊静态 UI。**在进入 V2 群聊 + Orchestrator 之前**，必须先在本阶段把两类「运行中需用户回应」的能力在 **单聊** 跑通：

1. **Approval**：Agent 请求写入文件、执行命令、调用工具等，需用户批准/拒绝。
2. **Choice（选项交互）**：Agent 弹出 A/B/C/D 或自定义输入，需用户选择后继续 **同一 run**。

本阶段 **不做** 群聊真实调度、Orchestrator、Provider；群聊仍保持 V1 静态 UI。但契约与数据模型 **自设计起预留** `conversation_agent_id`、`orchestrator_task_id`，以便 V2 直接复用。

**下一主线**：本阶段验收通过后，再启动 `V2-群聊与Orchestrator实施计划.md`。

## 2. 为什么要放在 V1 与 V2 之间

| 问题 | 若直接做 V2 | 本阶段做法 |
| --- | --- | --- |
| 子 Agent 写代码需审批 | Orchestrator + 审批 + 多 Agent 同时改，联调面过大 | 单聊先跑通 pause → 用户决定 → resume |
| AskUserQuestion / 选项 | 群聊里不知挂在哪个气泡 | 单聊 inline 卡片先定 UI 与 API |
| 当前 `process-runner` 一次性 stdin | V2 必然踩坑 | 先改 **可暂停 run** 与 **交互回传** |
| 群聊 UI 差异 | 无共用后端则重复造轮子 | 同一套 `AgentInteraction` + 按 `mode` 分 UI |

## 3. 产品边界

### 3.1 必做（V1.5）

- 统一 **`AgentInteraction`** 契约（approval + choice 共用骨架）。
- 单聊：**Approval** 仅在消息流 running 气泡下 inline 操作（右栏不重复卡片，见 `prototypes/v2/approval-ui.html` Tab①）。
- 单聊：**Choice** 在消息流 inline 展示选项（**单选** + 可选自定义文本；V1.5 不做多选 UI）。
- 用户回应后 **同一 `agent_run` 继续**，不强制新开 run。
- SSE：`interaction_requested` / `interaction_resolved`（或分类型事件，见 §6）。
- 至少 **1 个** adapter 跑通 Approval；至少 **1 个** adapter 跑通 Choice（可同为 Claude Code，Codex 时间允许再接）。
- run 状态：`running` → `awaiting_interaction` → `running` → `done` / `failed` / `cancelled`。

### 3.2 V1.5 不做

- 群聊真实消息 API、Orchestrator 调度（仍 400 / mock）。
- 群聊 Approval / Choice **UI**（V2.4 再接；本阶段只保证 API/字段可挂 `conversation_agent_id`）。
- Provider 表、Planner、Skill、自建 Agent。
- 审批/选项的批量自动策略、Orchestrator 代点。

### 3.3 V2 预留（本阶段就要设计进 schema / 类型）

```ts
type AgentInteractionContext = {
  conversationId: string;
  runId: string;
  messageId: string;              // 挂 inline UI 的 assistant 消息
  agentId: string;
  conversationAgentId?: string | null;  // V2 群聊子 Agent
  orchestratorTaskId?: string | null; // V2 task
};
```

群聊 UI 分工（V2 实现，本阶段文档定稿）：

| 交互类型 | 单聊（V1.5 实现） | 群聊（V2 实现） |
| --- | --- | --- |
| Approval | 消息流 inline 卡片 | 右栏按 Agent/task 聚合；消息流 system + badge |
| Choice | 消息流 inline 选项卡片 | 右栏聚合或锚定到对应 Agent 气泡（实现时二选一，共用 API） |

## 4. 与现状差距

当前代码 **前后端均无** Approval / Choice：

- `AgentEvent` 仅有 `text_delta`、产物、done/error。
- `runProcess` 写 stdin 后立即 `end()`，无法 mid-run 回传。
- Claude 使用 `--permission-mode plan`，未接 SDK `canUseTool`。
- 无交互表、无 respond API、无 inline 卡片组件。

本阶段需改 **适配器契约 → run 生命周期 → DB → API → SSE → 单聊 UI** 全链。

## 5. 模块设计

### 5.1 目录增量

```text
lib/
├── interactions/
│   ├── types.ts          # ApprovalPayload | ChoicePayload | InteractionDecision
│   ├── service.ts        # create / resolve / listPending
│   └── run-bridge.ts     # 挂起 run、唤醒 adapter（Promise / 队列）
├── adapters/
│   └── types.ts          # 扩展 AgentEvent + AdapterControl
app/api/
├── interactions/
│   └── [interactionId]/
│       └── respond/route.ts
components/chat/
├── InteractionApprovalCard.tsx
└── InteractionChoiceCard.tsx
```

### 5.2 统一交互模型

```ts
type InteractionKind = "approval" | "choice";

type AgentInteraction = {
  id: string;
  kind: InteractionKind;
  status: "pending" | "approved" | "rejected" | "answered" | "expired" | "cancelled";
  conversationId: string;
  runId: string;
  messageId: string;
  agentId: string;
  conversationAgentId?: string | null;
  orchestratorTaskId?: string | null;
  payload: ApprovalPayload | ChoicePayload;
  response?: InteractionDecision | null;
  createdAt: number;
  resolvedAt?: number | null;
};

type ApprovalPayload = {
  action: "write_file" | "run_command" | "tool_use" | "network" | string;
  summary: string;
  path?: string;
  command?: string;
  risk?: string;
};

type ChoicePayload = {
  prompt: string;
  options: Array<{ id: string; label: string; description?: string }>;
  allowCustom?: boolean;
  // multiSelect 保留在类型上供未来扩展；V1.5 UI 不实现多选
};

type InteractionDecision =
  | { kind: "approval"; approved: boolean }
  | { kind: "choice"; selectedOptionIds: string[]; customText?: string };
```

### 5.3 Adapter 扩展

`AgentEvent` 增量：

```ts
| { type: "interaction_required"; interaction: Omit<AgentInteraction, "id" | "status" | "response"> }
| { type: "interaction_resolved"; interactionId: string }  // adapter 内部 ack，可选
```

Adapter 需支持 **挂起**（非结束 run）：

- 优先路径：**Claude Agent SDK** `canUseTool` + AskUserQuestion 回调（Approval + Choice 分流）。
- 过渡路径：若 CLI 无法 mid-run，文档记录限制，该 adapter 在 V1.5 标为 partial，不阻塞另一 adapter 验收。
- `run-bridge`：在 `runs.ts` 内维护 `pendingInteraction` Map；收到 `interaction_required` 时 run → `awaiting_interaction`，SSE 推送；`respond` 后 resolve Promise，adapter 继续。

`process-runner` 增量（按需）：

- 长驻 stdin 模式或 SDK 替代一次性 spawn；**禁止**在仍有 pending interaction 时 `stdin.end()`。

### 5.4 单聊 UI

**Approval**（已定稿）：

- 挂在触发该 run 的 assistant 气泡下方 `InteractionApprovalCard`。
- 右栏 Agent 状态可显示 `awaiting_approval`，**不**放第二张卡片。

**Choice**：

- 同位置 `InteractionChoiceCard`（挂在 running 气泡下方，与 Approval 并列规则）。
- **视觉**：蓝色系卡片（Approval 为橙色系），避免两种交互混淆。
- **结构**：`prompt` 问题文案 + 选项列表（每项：`letter` 可选 A/B/C/D + `label` + 一行 `description`）+ 可选自定义输入 + 提交区。
- **单选**：点击一项即 `POST respond`，卡片从 UI **移除**，同一 run 继续流式输出。
- **自定义**（`allowCustom: true`）：「其他」+ textarea +「提交选择」；提交后同样移除卡片。
- **不做「已回答」只读 UI**：用户选择只写入 `agent_interactions.response_json` 并喂回 adapter，消息流不保留「已选择 B」卡片。
- **不做多选 UI**：Claude/Codex AskUserQuestion 典型为单选；若 adapter 报多选，V1.5 按单选处理或记 TOFIX。
- 右栏仅 `awaiting_choice` 状态，**不**重复选项 UI。UI 示意见 `docs/design/prototypes/v2/choice-ui.html`。

Composer：交互 pending 时允许输入被禁用或仅允许「取消 run」；与 Approval 一致。

### 5.5 Orchestrator 关系（V2）

V1.5 **不实现** Orchestrator；仅在类型上预留 `orchestratorTaskId`。V2 Scheduler 规则：

- task 进入 `awaiting_interaction` 时 pause 该 task，不代批、不代选。
- 用户 resolve 后 Invoker 同一 run 继续。

## 6. 数据模型

```text
agent_interactions
├── id
├── kind                    (approval | choice)
├── status
├── conversation_id
├── run_id
├── message_id
├── agent_id
├── conversation_agent_id   (nullable, V2)
├── orchestrator_task_id    (nullable, V2)
├── payload_json
├── response_json
├── created_at
└── resolved_at

agent_runs.status 增量：awaiting_interaction
messages.status  可反映：running | done | error | cancelled（交互态由 interaction 表 + run 状态表达）
```

## 7. API

```text
GET  /api/conversations/:id/interactions?status=pending   # 刷新恢复；V2 群聊右栏也用
POST /api/interactions/:interactionId/respond
     body: InteractionDecision
```

响应后：

- 更新 `agent_interactions`；
- 唤醒 `run-bridge`；
- SSE `interaction_resolved`；
- adapter 继续 → 后续仍走 `message_delta` 等既有事件。

## 8. SSE 增量

```text
interaction_requested   { interaction: AgentInteraction }
interaction_resolved    { interactionId, status, response? }
```

单聊与 V2 群聊 **共用事件名**；前端按 `conversation.mode` 与 `conversationAgentId` 决定渲染位置。

## 9. 实施阶段

### Phase V1.5.0 契约与 run 桥接

- `lib/interactions/types.ts`、`agent_interactions` 表 migration。
- 扩展 `AgentEvent`、`stream-bus`、run 状态机（`awaiting_interaction`）。
- `run-bridge` 骨架 + 单测（mock adapter 发 `interaction_required` → respond → done）。
- 更新 `API_CONTRACT.md` 本节。

验收：mock adapter 单聊 E2E（无真实 CLI 也可）能 pause/resume。

### Phase V1.5.1 Approval（单聊）

- Claude Code（或首个可用 adapter）接 Approval 来源（SDK `canUseTool` 优先）。
- `InteractionApprovalCard` + `POST .../respond`。
- SSE + 刷新恢复 pending。

验收：单聊触发写文件/命令审批 → inline 批准 → 同一 run 继续 → 拒绝则 run failed。UI 符合 `approval-ui.html` 单聊 Tab。

### Phase V1.5.2 Choice（单聊）

- 同一 adapter 或第二个 adapter 接 AskUserQuestion / 等价 JSON。
- `InteractionChoiceCard`（选项 + 可选自定义）。
- 同一 run resume。

验收：Agent 弹出选项 → 用户点选或填自定义 → 同一 run 继续；刷新后 pending 可恢复操作。

### Phase V1.5.3 收口与 V2 门禁

- typecheck / build；单聊主链路回归（发消息、stop、regenerate 不退化）。
- 文档：在 V2 计划 §0 勾选「V1.5 已通过」。
- 问题写入 `TOFIX.md`。

**V2 启动门禁**：Phase V1.5.1 + V1.5.2 验收通过；否则 V2 不做群聊 implement_review 真实写盘 Demo。

## 10. 验收标准（V1.5 总表）

- 单聊 Approval inline 可用；右栏不重复审批卡片。
- 单聊 Choice inline 可用；非「只能新开一条消息假装回复」。
- 同一 `run_id` 在交互 resolve 后继续执行。
- `agent_interactions` 含可空 `conversation_agent_id`、`orchestrator_task_id`。
- SSE 与 API 对群聊模式 **无 hardcode single**（V2 只加 UI，不改契约）。
- 群聊仍不接真实消息（V1 边界保持）。
- V1 单聊发消息 / 流式 / stop / 产物 / 刷新恢复不退化。
