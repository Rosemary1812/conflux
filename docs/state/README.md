# 状态池

本文件夹不记录项目主线进度。主线阶段和优先级放在根目录 `roadmap.md`，产品和技术设计放在 `docs/design/`。

`docs/state/` 只用于管理开发中后期发现的零散小问题、小优化和临时修补项，例如 UI 细节、体验瑕疵、非阻塞 bug、技术债、后续清理点。

## 文件说明

- `TODO.md`：主线之外的小目标、小优化、补充事项。
- `TOFIX.md`：已发现但未必马上修的 bug、回归、技术债和体验问题。

每个文件都分为两个部分：

- `待做`：尚未处理。
- `已做`：已经完成。条目标记完成后，应从 `待做` 移到 `已做`，不要另建 `DONE.md`。

## 条目格式

`TODO.md` 和 `TOFIX.md` 使用同一套字段：

```md
- 时间：YYYY-MM-DD HH:mm
  优先级：P1 / P2 / P3 / 待定
  所属范围：UI / API / 适配器 / Orchestrator / 文档 / 构建 / QA / 其他
  问题/目标：一句话描述要解决的问题或目标
  解决方案：计划采用的处理方式；未确定时写“待定”
  涉及修改文件：`path/to/file`、`path/to/other-file`
  验收标准：完成后如何判断可接受
```

## 写入规则

- 不把 P0/P1/P2 主线任务拆进这里，除非它是主线外延伸出来的小修补项。
- 如果条目来自 review、QA 或人工判断，并带有 P1/P2/P3 等评级，应原样保留到 `优先级` 字段；没有评级时写“待定”。
- 不粘贴完整聊天记录。
- 不把已经完成的条目留在 `待做`，完成后移动到同文件的 `已做`。
- 如果问题需要进入主线规划，应提升到 `roadmap.md` 或 `docs/design/`，而不是长期停留在这里。
