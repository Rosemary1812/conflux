# State TOFIX

管理当前项目的 `docs/state/TOFIX.md`。

## 使用边界

只处理 bug、回归、技术债、体验问题、QA 发现的问题、代码 review 后暂不立即修复的小问题。

不要把主线阶段任务写入 `TOFIX.md`。主线任务应留在 `roadmap.md` 或 `docs/design/ExecutePlan/`。

## 工作模式

### 1. 写入 TOFIX

当用户说“记录到 tofix”、“把这些 review 问题写入 TOFIX”、“调用 tofix 记录问题”时：

1. 读取 `docs/state/README.md`，确认条目格式。
2. 读取 `docs/state/TOFIX.md`，避免重复写入。
3. 从用户输入、review 结果或当前上下文中提取待修复问题。
4. 为每个问题生成一条 `待做` 条目。
5. 保留优先级评级，例如 `P1`、`P2`、`P3`。如果原始问题没有评级，不要强行编造；可以写 `优先级：待定`。
6. 写入 `## 待做` 下方，不要写到 `## 已做`。
7. 不粘贴完整 review 原文或聊天记录，只写可执行的问题摘要。

条目格式：

```md
- 时间：YYYY-MM-DD HH:mm
  优先级：P1 / P2 / P3 / 待定
  所属范围：UI / API / 适配器 / Orchestrator / 文档 / 构建 / QA / 其他
  问题/目标：一句话描述要解决的问题
  解决方案：计划采用的处理方式；未确定时写“待定”
  涉及修改文件：`path/to/file`、`path/to/other-file`
  验收标准：完成后如何判断可接受
```

### 2. 读取待修复项

当用户说“看看 tofix 里还有什么”、“读取未完成 tofix”、“列一下待修复项”时：

1. 读取 `docs/state/TOFIX.md`。
2. 只列出 `## 待做` 下的条目。
3. 按优先级和出现顺序整理。
4. 不自动修复，除非用户明确要求。

输出格式：

```md
当前 TOFIX 待做：

1. [P1][UI] 问题摘要
   涉及：`path`
   验收：...
2. [P2][API] 问题摘要
   涉及：`path`
   验收：...
```

### 3. 修复 TOFIX 条目

当用户要求“修复第 N 个”、“修复这些 P1”、“把某个 tofix 做掉”时：

1. 读取 `docs/state/TOFIX.md`。
2. 确认目标条目。
3. 阅读涉及文件和相关代码。
4. 只修复用户指定的条目，不扩大范围。
5. 完成后运行合理验证命令。
6. 将已完成条目从 `## 待做` 移动到 `## 已做`。
7. 可以在条目下补充：
   - `完成时间：YYYY-MM-DD HH:mm`
   - `验证结果：...`

## 注意事项

- 不要删除未完成条目。
- 不要把已完成条目留在 `待做`。
- 不要把主线 V1/V2/V3 任务拆进 `TOFIX.md`。
- 不要为了让测试通过而隐藏真实问题。
- 如果条目已经过期或不属实，先向用户说明证据，再按用户决定处理。
