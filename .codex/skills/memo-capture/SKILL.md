# Memo Capture

把一次讨论整理成 `docs/memo/` 下的结构化思考文档。

## 使用边界

用于沉淀思考，不用于记录主线进度或执行状态。

适合记录：

- 技术选型讨论
- 产品范围取舍
- 架构方案比较
- phase/version 策略思考
- 风险复盘
- 人与 AI 协作方式的沉淀

不适合记录：

- 完整聊天记录
- 具体待修复 bug（使用 `state-tofix`）
- 主线之外的小待办（使用 `state-todo`）
- 给下一个 Agent 继续工作的交接提示（使用 `project-handoff`）

## 写入位置

默认写入：

```txt
docs/memo/YYYY-MM-DD-HHmm-topic-slug.md
```

文件名规则：

- 时间使用本地时间。
- `topic-slug` 使用英文小写短横线。
- slug 应简短表达主题，例如：
  - `adapter-contract`
  - `v1-phase-splitting`
  - `docs-workflow`
  - `memo-state-skills`

文档标题可以使用中文。

## 工作流程

1. 读取 `docs/memo/README.md`，确认 memo 目录规则。
2. 根据当前对话提取讨论主题、背景、选项、取舍、结论和后续动作。
3. 不粘贴完整聊天记录。
4. 如果已有明确决策，状态写 `已决策`。
5. 如果只是阶段性倾向，状态写 `暂定`。
6. 如果需要后续实验或确认，状态写 `待验证`。
7. 如果只是保留思考，状态写 `仅记录思考`。
8. 如果讨论产生行动项，在“后续动作”中写明应进入哪里：
   - 主线计划：`roadmap.md` 或 `docs/design/ExecutePlan/`
   - 小优化：`docs/state/TODO.md`
   - bug/技术债：`docs/state/TOFIX.md`
9. 写入 `docs/memo/` 后，回复用户文件路径和简短摘要。

## Memo 模板

使用以下结构：

```md
# 标题

- 时间：YYYY-MM-DD HH:mm
- 类型：技术选型 / 产品取舍 / 架构讨论 / 实施策略 / 风险复盘 / 工作流沉淀 / 其他
- 相关范围：UI / API / 适配器 / Orchestrator / 文档 / 工作流 / 其他
- 状态：已决策 / 暂定 / 待验证 / 仅记录思考
- 相关文件：`path/to/file`、`path/to/other-file`

## 背景

这次讨论为什么发生，问题从哪里来。

## 核心问题

- 问题 1
- 问题 2
- 问题 3

## 讨论要点

- 要点 1
- 要点 2
- 要点 3

## 方案对比

| 方案 | 优点 | 缺点 | 适用条件 |
| --- | --- | --- | --- |
| 方案 A | ... | ... | ... |
| 方案 B | ... | ... | ... |

## 当前结论

本次讨论形成的结论。如果还没有结论，写“暂未决策”。

## 决策理由

为什么当前倾向这个方案，主要依据是什么。

## 后续动作

- [ ] 需要验证的点
- [ ] 需要补充到设计文档的点
- [ ] 需要进入 TODO/TOFIX 的点

## 不纳入范围

本次明确不处理、不展开或暂缓的内容。
```

## 质量要求

- 用中文输出正文。
- 保持结构化，但不要为了填模板编造内容。
- 没有方案对比时，可以写“本次没有形成多个备选方案”。
- 没有后续动作时，写“暂无”。
- 相关文件只列实际相关路径；不确定时写“暂无”。
- 讨论中的临时想法和最终结论要分开写。
