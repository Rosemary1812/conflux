# State TODO

管理当前项目的 `docs/state/TODO.md`。

## 使用边界

只处理主线之外的小目标、小优化、补充事项。

不要把 roadmap、version、phase 的主线开发任务写入 `TODO.md`。主线任务应写在 `roadmap.md` 或 `docs/design/ExecutePlan/`。

## 工作模式

### 1. 写入 TODO

当用户说“记录到 todo”、“把这个优化加入 TODO”、“调用 todo 记录一下”时：

1. 读取 `docs/state/README.md`，确认条目格式。
2. 读取 `docs/state/TODO.md`，避免重复写入。
3. 从用户输入或当前上下文中提取待办事项。
4. 为每个事项生成一条 `待做` 条目。
5. 保留优先级评级，例如 `P1`、`P2`、`P3`。如果原始事项没有评级，不要强行编造；可以写 `优先级：待定`。
6. 写入 `## 待做` 下方，不要写到 `## 已做`。

条目格式：

```md
- 时间：YYYY-MM-DD HH:mm
  优先级：P1 / P2 / P3 / 待定
  所属范围：UI / API / 适配器 / Orchestrator / 文档 / 构建 / QA / 其他
  问题/目标：一句话描述要完成的小目标或优化
  解决方案：计划采用的处理方式；未确定时写“待定”
  涉及修改文件：`path/to/file`、`path/to/other-file`
  验收标准：完成后如何判断可接受
```

### 2. 读取待办项

当用户说“看看 todo 里还有什么”、“读取未完成 todo”、“列一下待办”时：

1. 读取 `docs/state/TODO.md`。
2. 只列出 `## 待做` 下的条目。
3. 按优先级和出现顺序整理。
4. 不自动执行，除非用户明确要求。

输出格式：

```md
当前 TODO 待做：

1. [P1][UI] 目标摘要
   涉及：`path`
   验收：...
2. [P2][文档] 目标摘要
   涉及：`path`
   验收：...
```

### 3. 完成 TODO 条目

当用户要求“做第 N 个 todo”、“完成这些 P2 todo”、“把某个 todo 做掉”时：

1. 读取 `docs/state/TODO.md`。
2. 确认目标条目。
3. 阅读涉及文件和相关上下文。
4. 只完成用户指定的条目，不扩大范围。
5. 完成后运行合理验证命令。
6. 将已完成条目从 `## 待做` 移动到 `## 已做`。
7. 可以在条目下补充：
   - `完成时间：YYYY-MM-DD HH:mm`
   - `验证结果：...`

## 注意事项

- 不要删除未完成条目。
- 不要把已完成条目留在 `待做`。
- 不要把主线 phase 任务写进 `TODO.md`。
- 如果待办事项已经不适用，先说明原因，再按用户决定处理。
